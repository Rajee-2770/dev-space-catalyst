02 · Stack
STACK

Real-World Applications

Where stacks power the systems you use every day

💻
OS — Function Call Stack
Every function call pushes a stack frame (return address, local vars, params). On return, frame is popped. Stack overflow = too deep recursion exceeds stack memory.
🌐
Browser — Back/Forward Navigation
Two stacks: Back Stack (pages visited) and Forward Stack (pages after pressing back). Navigate back = pop from back, push to forward.
⌨
Editor — Undo/Redo System
Each action pushed to undo stack. Ctrl+Z pops undo, pushes to redo stack. New edit clears redo stack. Used in VS Code, Photoshop, etc.
🔢
Compiler — Expression Parsing
Infix to postfix conversion, parenthesis matching, syntax tree building — all use stacks. Shunting-yard algorithm relies entirely on an operator stack.
📱
Android / iOS Activity Stack: Each screen pushed onto the activity back stack. Press back = pop current, reveal previous. Task kill = clear entire stack.

05 · Queue
QUEUE

Real-World Applications

Queues are the backbone of asynchronous and distributed systems

💻
OS — Process Scheduling
Ready Queue: processes waiting for CPU time. Scheduler dequeues next process, runs it, re-enqueues if time slice expires (Round Robin). Multiple priority queues in modern OS.
📨
Message Queues — Kafka, RabbitMQ
Producers enqueue messages, consumers dequeue them. Decouples services, handles traffic spikes, enables retry logic. At-least-once vs exactly-once delivery guarantees.
🎨
Print Spooler & I/O Buffers
Print jobs queued in FIFO order. Keyboard buffer stores keystrokes faster than CPU processes them. Network packet buffers use circular queues to prevent data loss.
📈
BFS & Level-Order Traversal
Breadth-First Search uses a queue to explore nodes level by level. Tree level-order traversal, shortest path in unweighted graphs, web crawlers — all queue-based.
📱
Android Handler/Looper: UI thread has a MessageQueue. Handler posts Runnables to the queue. Looper dequeues and executes them one at a time on the main thread — this is how Android UI updates work.

06 · Circular Queue
CIRCULAR QUEUE

Ring Buffer — No Wasted Space

Front and rear wrap around using modulo arithmetic

CIRCULAR BUFFER (capacity=6)
Rear wraps to index 0 after reaching end — no shifting needed
next = (current + 1) % capacity
isEmpty: size == 0  ·  isFull: size == capacity
Used In
Network packet buffers · Audio streaming buffers · OS keyboard input buffer · Producer-consumer with bounded buffer
Why not plain array queue?
Linear queue wastes space: dequeued slots at front become unusable. Shifting costs O(n). Circular queue reuses slots — O(1) always.

From Arrays topic (Slide 1-2)

12 · Decision Framework
FRAMEWORK

Which ADT Should You Use?

Follow this decision tree for any scenario

1.
Need LIFO (undo, backtracking, recursion)?
→ Stack (use ArrayDeque in Java)
2.
Need FIFO (scheduling, BFS, buffering)?
→ Queue (ArrayDeque or LinkedList)
3.
Need insert/remove from both ends?
→ Deque (sliding window, palindrome check)
4.
Need ordering by priority, not insertion order?
→ Priority Queue (Dijkstra, task scheduling, top-K)
5.
Fixed-size buffer with wrap-around?
→ Circular Queue / Ring Buffer (I/O, streaming)
6.
Need thread-safe queue?
→ ConcurrentLinkedQueue (non-blocking) or BlockingQueue (producer-consumer)
"Restrict access patterns, and simple structures become powerful."

STACKHARD
Q1: How does the OS handle nested function calls using the call stack? What happens during a stack overflow?
HINT
Think about stack frames, return addresses, and memory limits per thread.
ANSWER
Each function call pushes a stack frame containing: return address, parameters, local variables, saved registers. Nested calls stack frames on top.

On return, the frame is popped and execution resumes at the return address. If recursion has no base case or nesting is too deep, frames exceed the stack memory limit (typically 1-8 MB per thread) → StackOverflowError.

Real scenario: A recursive JSON parser processing deeply nested objects (10,000+ levels) will overflow. Solution: convert to iterative with an explicit stack on the heap (no size limit).
↓
STACKSENIOR
Q2: Design an Undo/Redo system for a text editor. Which data structures and why?
HINT
Two stacks. What happens to the redo stack when a new action is performed?
ANSWER
Two stacks: undoStack and redoStack.

• New action: push to undoStack, clear redoStack (forward history invalidated)
• Undo (Ctrl+Z): pop from undoStack, push to redoStack, reverse the action
• Redo (Ctrl+Y): pop from redoStack, push to undoStack, re-apply the action

Each entry stores the command object (Command Pattern): action type, position, old text, new text. This allows both undo and redo to be O(1).

Optimization: Group rapid keystrokes into batches (e.g., undo a whole word, not letter-by-letter). Cap stack size to limit memory.

STACKHARD
Q3: How would you validate balanced parentheses including {}, [], ()? What is the time and space complexity?
HINT
Push opening brackets, pop on closing brackets, check if they match.
ANSWER
Use a stack. For each character:
• Opening bracket → push to stack
• Closing bracket → pop from stack, check it matches
• If stack empty when closing or mismatch → invalid
• After all chars, stack must be empty

Time: O(n) — single pass. Space: O(n) — worst case all opening brackets.

Real-world: Every compiler uses this for syntax validation. JSON parsers validate nested braces. HTML validators check tag matching. IDEs highlight mismatched brackets in real-time.

QUEUESENIOR
Q4: Explain how the OS Round Robin scheduler uses a queue. What happens when a process exceeds its time quantum?
HINT
Think about dequeue from front, execute for quantum, re-enqueue if not finished.
ANSWER
Round Robin: All runnable processes sit in a FIFO ready queue.

1. Dequeue front process, assign CPU for time quantum (e.g., 10ms)
2. If process completes before quantum → remove from system
3. If process exceeds quantum → preempt, save context, enqueue at rear
4. Repeat

Trade-off: Small quantum = fairer but more context switches (overhead). Large quantum = fewer switches but poor response time. Typical: 10-100ms.

Real OS: Linux CFS doesn't use pure RR but a weighted fair-share queue using a red-black tree. Android uses CFS + priority boosting for foreground apps.

QUEUEHARD
Q5: Implement a Queue using two Stacks. What is the amortized time complexity of enqueue and dequeue?
HINT
One stack for input, one for output. Transfer only when output stack is empty.
ANSWER
Two stacks: inStack and outStack.

• enqueue(x): push x to inStack → O(1)
• dequeue(): if outStack empty, pop ALL from inStack and push to outStack. Then pop from outStack.

Worst-case dequeue is O(n) (when transfer happens). But each element is transferred at most once, so amortized O(1) for both operations.

Real-world: This pattern appears in functional programming (persistent queues) and in systems where you batch-process accumulated items (e.g., log aggregation: write to buffer stack, flush to processing stack).

STACKSYSTEM DESIGN
Q6: How does the Android Activity back stack work? What happens when you press the back button?
HINT
Think about task, back stack, and activity lifecycle.
ANSWER
Android manages activities in a back stack within each task:

• Start activity: pushed onto the stack (onCreate → onResume)
• Press back: current activity popped (onPause → onStop → onDestroy), previous activity resumes
• Stack empty: task finishes, returns to home/launcher

Launch modes modify stack behavior:
- singleTop: reuses top if same activity
- singleTask: clears everything above it
- singleInstance: own task with single activity

Fragment back stack works similarly within an activity. Navigation component manages a fragment back stack internally using the same LIFO pattern.

QUEUESYSTEM DESIGN
Q7: How does Android's Handler/Looper/MessageQueue work? Why is it a queue?
HINT
One thread, one Looper, one MessageQueue. Handler posts messages.
ANSWER
Architecture:
• MessageQueue: FIFO queue of Message objects (sorted by timestamp)
• Looper: infinite loop that dequeues from MessageQueue and dispatches
• Handler: posts Messages or Runnables to the queue

Main thread Looper runs forever. UI updates, touch events, lifecycle callbacks — all posted as messages. This is why you can't do network calls on the main thread (blocks the queue).

Why queue? Serializes access to the UI thread — no locks needed. All updates happen in order. postDelayed() inserts at the right position by timestamp. This is a priority queue by scheduled time under the hood.

CIRCULAR QUEUESENIOR
Q8: Design a circular buffer for a real-time audio streaming application. What happens when the producer is faster than the consumer?
HINT
Ring buffer with read/write pointers. Consider overflow policy: drop oldest or block producer.
ANSWER
Circular buffer with fixed capacity (e.g., 4096 audio frames):

• Write pointer (producer) advances on new data
• Read pointer (consumer) advances on consumption
• Both wrap around via modulo

Overflow policies:
1. Overwrite oldest (lossy): write pointer overwrites unread data — used in audio/video where stale frames are useless
2. Block producer (lossless): producer waits until consumer catches up — used in file I/O
3. Drop new data: reject incoming data if full — used in network rate limiting

Lock-free variant: Single producer, single consumer can use atomic read/write pointers with no locks (SPSC queue). Used in Linux kernel's kfifo.

PRIORITY QUEUEHARD
Q9: How does Dijkstra's algorithm use a priority queue? What happens if you use a regular queue instead?
HINT
Greedy: always expand the closest unvisited node. Regular queue gives BFS (unweighted shortest path only).
ANSWER
Dijkstra's: Min-heap priority queue keyed by tentative distance.

1. PQ starts with source node (dist=0)
2. Poll min-distance node, relax all its edges
3. If shorter path found, insert/update neighbor in PQ
4. Repeat until PQ empty or target reached

Time: O((V+E) log V) with binary heap.

Regular queue gives BFS: works for unweighted graphs only (all edges weight=1). For weighted graphs, BFS may explore a longer-distance node before a shorter one, giving wrong answers.

Real-world: Google Maps, network routing (OSPF protocol), game pathfinding (A* is Dijkstra + heuristic).

DEQUESENIOR
Q10: Solve "Sliding Window Maximum" using a deque. Why is a naive approach O(n*k) but deque gives O(n)?
HINT
Maintain a monotonically decreasing deque of indices. Remove from front if out of window, from back if smaller than current.
ANSWER
Naive: For each window position, scan k elements for max → O(n*k).

Monotonic Deque (O(n)):
1. Deque stores indices in decreasing order of values
2. For each new element i:
  • Remove from back all indices whose values ≤ arr[i] (they'll never be max)
  • Add i to back
  • Remove from front if index is outside window (i - front ≥ k)
  • Front of deque is the max for current window

Each element enters and leaves the deque at most once → O(n) total.

Real-world: Rolling max/min in stock analysis, network throughput monitoring, image processing (max-pooling in CNNs).

STACKHARD
Q11: Convert infix expression "3 + 4 * 2 / (1 - 5)" to postfix using Shunting-Yard algorithm. Explain each step.
HINT
Use an operator stack. Pop operators with higher or equal precedence. Parentheses act as barriers.
ANSWER
Shunting-Yard (Dijkstra's algorithm):

Token: 3 → output: [3]
Token: + → stack: [+]
Token: 4 → output: [3,4]
Token: * → stack: [+,*] (* > + so push)
Token: 2 → output: [3,4,2]
Token: / → pop * (same prec), push /. stack: [+,/]. output: [3,4,2,*]
Token: ( → stack: [+,/,(]
Token: 1 → output: [3,4,2,*,1]
Token: - → stack: [+,/,(,-]
Token: 5 → output: [3,4,2,*,1,5]
Token: ) → pop until (: pop -. output: [3,4,2,*,1,5,-]
End: pop remaining. output: 3 4 2 * 1 5 - / +

Used by: Compilers, calculators, spreadsheet formula engines.

QUEUESYSTEM DESIGN
Q12: Design a rate limiter using a queue. How does the sliding window approach work?
HINT
Queue stores timestamps of requests. Dequeue expired ones, check size against limit.
ANSWER
Sliding Window Log (Queue-based):

1. Queue stores timestamps of each request
2. On new request at time T:
  • Dequeue all entries older than T - windowSize
  • If queue.size() < maxRequests: allow, enqueue T
  • Else: reject (429 Too Many Requests)

Time: O(1) amortized. Space: O(maxRequests).

Real-world: API gateways (Nginx, Kong), Twitter API limits, Discord bot rate limits. For distributed systems, use Redis sorted sets (timestamp as score) — essentially a distributed queue.

Trade-off: Fixed window counter is simpler but has boundary burst issue. Sliding window log is precise but uses more memory.

STACKSENIOR
Q13: Design a Min Stack that supports push, pop, top, and getMin — all in O(1) time.
HINT
Use an auxiliary stack that tracks the minimum at each level.
ANSWER
Two stacks: mainStack and minStack.

• push(x): push to mainStack. Push min(x, minStack.peek()) to minStack
• pop(): pop from both stacks
• top(): mainStack.peek()
• getMin(): minStack.peek() → O(1)

Space optimization: Only push to minStack when x ≤ current min. On pop, only pop minStack if value equals current min. Saves space when min changes rarely.

Real-world: Monitoring systems tracking minimum latency over a sliding window. Stock trading: track minimum price seen so far for maximum profit calculation.

QUEUEHARD
Q14: How do message queues (Kafka, RabbitMQ) handle the producer-consumer problem? What guarantees do they provide?
HINT
Think about decoupling, buffering, delivery guarantees (at-most-once, at-least-once, exactly-once).
ANSWER
Message Queue pattern:
• Producer enqueues messages (fire-and-forget or with ack)
• Consumer dequeues and processes
• Queue decouples them: producer doesn't wait for consumer

Delivery guarantees:
1. At-most-once: message may be lost (fastest, no retry)
2. At-least-once: message may be duplicated (consumer must be idempotent)
3. Exactly-once: hardest, requires transactional writes (Kafka supports this)

Kafka specifics: Partitioned queue (topic). Consumer groups for parallel processing. Offset tracking for replay. Messages persisted to disk — not deleted on consume.

Real-world: Order processing (e-commerce), event sourcing, log aggregation, microservice communication.

PRIORITY QUEUESYSTEM DESIGN
Q15: Design a task scheduler for an OS where processes have different priorities. How do you prevent starvation?
HINT
Priority queue for scheduling. Starvation happens when low-priority tasks never execute. Think aging.
ANSWER
Priority Queue Scheduler: Always dequeue highest-priority process.

Starvation problem: Low-priority process waits forever if high-priority processes keep arriving.

Solutions:
1. Aging: Gradually increase priority of waiting processes. After enough time, even lowest priority becomes highest → guaranteed execution
2. Multilevel Feedback Queue: Process starts at high priority. If it uses full quantum, demote to lower queue. I/O-bound processes stay high.
3. Fair-share scheduling: Guarantee minimum CPU % per group (Linux CFS)

Real OS example: Linux nice values (-20 to 19). Real-time priorities (0-99) preempt all normal processes. SCHED_FIFO for hard real-time (robots, medical devices).

STACKHARD
Q16: Given a histogram (array of bar heights), find the largest rectangle area. What data structure makes this O(n)?
HINT
Monotonic increasing stack of indices. Pop when current bar is shorter than stack top.
ANSWER
Monotonic Stack approach (O(n)):

Maintain a stack of indices with increasing heights:
1. For each bar i: while stack top height > heights[i], pop index p
2. Width = i - stack.peek() - 1 (or i if stack empty)
3. Area = heights[p] * width. Track max area.
4. Push i to stack

After loop, process remaining stack entries (treat end as height=0).

Why O(n)? Each index pushed and popped at most once → 2n operations total.

Naive: For each bar, expand left/right to find max rectangle → O(n²).

Extends to: Maximal rectangle in a binary matrix (apply per row as histogram). Used in image processing and layout optimization.

QUEUESENIOR
Q17: How does BFS use a queue to find the shortest path in an unweighted graph? Why won't DFS (stack) work?
HINT
BFS explores level by level. DFS dives deep first — may find a longer path before the shortest one.
ANSWER
BFS guarantees shortest path in unweighted graphs:

1. Enqueue source with distance 0
2. Dequeue node, enqueue all unvisited neighbors with distance+1
3. First time a node is visited = shortest distance (FIFO ensures level-order)

DFS fails because: It follows one path deeply. May reach the target via a long detour first. Would need to explore ALL paths and compare — exponential.

Key insight: Queue's FIFO property guarantees nodes at distance d are all processed before distance d+1. Stack's LIFO doesn't preserve this ordering.

Real-world: Social network "degrees of separation", maze solving, web crawlers (breadth-first crawling for important pages), network broadcast (flooding).

CIRCULAR QUEUESYSTEM DESIGN
Q18: How does a network router use circular buffers for packet queuing? What happens during congestion?
HINT
Ingress/egress queues. Tail drop vs RED (Random Early Detection). Buffer bloat.
ANSWER
Router packet queuing:
• Incoming packets enqueued in circular buffers (per-port or per-flow)
• Scheduler dequeues and forwards based on routing table

Congestion handling:
1. Tail Drop: buffer full → drop new packets. Simple but causes TCP synchronization (many flows back off simultaneously)
2. RED (Random Early Detection): start randomly dropping packets BEFORE buffer is full. Signals TCP to slow down gradually
3. WRED (Weighted RED): different drop probabilities per traffic class (QoS)

Buffer bloat: Too-large buffers cause excessive latency without dropping packets. Modern fix: CoDel (Controlled Delay) algorithm monitors per-packet sojourn time in the queue.

All implemented as circular queues for O(1) enqueue/dequeue with zero allocation overhead.

STACKSENIOR
Q19: Explain how DFS uses a stack (explicit or call stack) for graph traversal. When would you prefer explicit stack over recursion?
HINT
Recursive DFS uses call stack implicitly. Deep graphs risk stack overflow. Explicit stack uses heap memory.
ANSWER
DFS with explicit stack:
1. Push source node
2. Pop top, mark visited
3. Push all unvisited neighbors
4. Repeat until stack empty

Recursive DFS: Same logic, but function call stack handles push/pop automatically.

Prefer explicit stack when:
• Graph depth may exceed call stack limit (10,000+ nodes) → StackOverflow risk
• Need to control traversal order precisely
• Need to convert to iterative for performance (no function call overhead)
• Language lacks tail-call optimization (Java, Python)

Real-world: Topological sort (build systems like Make/Gradle), cycle detection, solving puzzles (Sudoku backtracking), compiler register allocation.

PRIORITY QUEUESENIOR
Q20: You have a stream of events arriving in real-time. Design a system to always know the top-K most frequent events. Which data structures?
HINT
HashMap for frequency counting + min-heap of size K. Why min-heap and not max-heap?
ANSWER
HashMap + Min-Heap (size K):

1. HashMap: event → frequency count (O(1) update per event)
2. Min-Heap of size K: stores the top-K elements by frequency

For each event:
• Update frequency in HashMap
• If element already in heap: update its frequency (sift up/down)
• If heap.size() < K: insert
• Else if frequency > heap.peek(): poll min, insert current

Why min-heap? The root is the smallest of top-K. Easy to check if new element belongs (compare with min). Max-heap of size K would require finding the min to evict — O(K).

Time: O(n log K) for n events. Space: O(n) for HashMap + O(K) for heap.

Real-world: Trending hashtags (Twitter), top search queries (Google), most active users, network anomaly detection.
