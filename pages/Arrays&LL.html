10 · Decision Framework
FRAMEWORK

Which Structure Should You Use?

Follow this decision tree for any scenario

1.
Need O(1) random access by index?
→ Size known → Array | Size varies → ArrayList
2.
Frequent insert/delete at head?
→ Need backward traversal? → Doubly LL | No → Singly LL
3.
Cyclic / round-robin pattern?
→ Circular Linked List
4.
O(1) delete by reference needed?
→ Doubly LL + HashMap (LRU pattern)
5.
Cache performance critical?
→ Array — contiguous memory always wins
6.
None of the above?
→ ArrayList — safest general-purpose default
"No best data structure — only the best for your problem."

ARRAYHARD
Q1: You insert n elements one by one at index 0 of an array. What is the total time complexity?
HINT
Each insertion at index 0 shifts ALL existing elements right by 1 position.
ANSWER
1st insert: 0 shifts. 2nd: 1 shift. 3rd: 2 shifts... nth: (n-1) shifts.

Total = 0+1+2+...+(n-1) = n(n-1)/2 = O(n²).

This is why ArrayList.add(0, val) in a loop is an anti-pattern — use LinkedList or ArrayDeque instead.
↓
ARRAYSENIOR
Q2: Two O(n) algorithms — one uses Array, one uses LinkedList. The Array version runs 50× faster. Why?
HINT
Think about CPU hardware, not Big-O notation.
ANSWER
CPU cache lines (64 bytes). Array elements sit contiguously → CPU prefetcher loads the next cache line automatically → ~99% cache hits.

LinkedList nodes are scattered across the heap → every node access is a potential cache miss (~100 cycle penalty each).

For 1M elements: Array gets streaming prefetch. LL gets pointer-chasing. Same O(n), vastly different wall-clock time. This is called spatial locality.
↓
ARRAYLISTHARD
Q3: After inserting 1000 elements from capacity 1 (2× growth), how many total element copies occurred?
HINT
Copies happen at each resize: capacity 1, 2, 4, 8... Sum the geometric series.
ANSWER
Resizes at: 1+2+4+8+16+32+64+128+256+512 = 1023 copies.

For 1000 inserts → ~1 copy/insert average → O(1) amortized.
Proof: geometric series sum = 2^(k+1) - 1 where 2^k ≤ n. Always < 2n.
Total work for n inserts = O(n) → per-insert = O(1) amortized.

ARRAYLISTSENIOR
Q4: ArrayList<Integer> uses 5× more memory than int[]. Explain exactly why.
HINT
Consider: object header, autoboxing, pointer size, padding.
ANSWER
int[]: 4 bytes per element.

ArrayList<Integer>: each element is an Integer object:
• 12B object header + 4B int value + 4B padding = 16B per Integer
• Plus 8B reference (pointer) in the backing Object[] array
• = ~24 bytes per element vs 4 bytes = 6× overhead

Plus unused capacity slots waste additional memory. This is why numeric-heavy code uses primitive arrays or Eclipse Collections.

SINGLY LLHARD
Q5: Reverse a Singly Linked List in-place with O(1) extra space. Walk through the algorithm.
HINT
Use three pointers: prev, current, next. Re-point each arrow backward.
ANSWER
prev = null, curr = head while (curr != null):   next = curr.next   curr.next = prev   prev = curr   curr = next head = prev

Time: O(n), Space: O(1). Each node visited exactly once. You must save curr.next BEFORE overwriting it. LeetCode #206.
↓
SINGLY LLSENIOR
Q6: Detect a cycle in a linked list. Find the cycle's starting node. Prove why it works.
HINT
Floyd's Tortoise and Hare — two pointers at different speeds.
ANSWER
Detection: slow moves 1 step, fast moves 2 steps. If cycle exists, they meet inside it. If fast hits NULL → no cycle. O(n) time, O(1) space.

Find start: After meeting, reset one pointer to head. Move both at speed 1. They meet at cycle start.

Proof: Let d = distance head→cycle-start, k = meeting point from cycle start, C = cycle length. When they meet: 2(d+k) = d+k+mC → d = mC−k. So moving d steps from meeting point lands at cycle start.
↓
SINGLY LLSENIOR
Q7: Find the middle node of a linked list in ONE pass without counting nodes first.
HINT
If one pointer runs twice as fast as the other...
ANSWER
Slow-fast pointer:
slow = head, fast = head while (fast != null && fast.next != null):   slow = slow.next   fast = fast.next.next return slow

When fast reaches the end, slow is at the midpoint. O(n) time, O(1) space.

DOUBLY LLSENIOR
Q8: Design an LRU Cache with O(1) get and O(1) put. Why is DLL + HashMap optimal?
HINT
What operations does LRU need? O(1) lookup, O(1) move-to-front, O(1) evict-from-tail.
ANSWER
HashMap<Key, DLLNode> for O(1) key→node lookup.
Doubly Linked List ordered by recency: head = most recent, tail = least recent.

get(key): HashMap O(1) → detach node O(1) → move to head O(1).
put(key,val): If exists, update + move to head. If new: insert at head. If over capacity: remove tail O(1) + delete from HashMap O(1).

Why not SLL? Detaching needs prev → O(n) to find it. Why not Array? Move-to-front shifts → O(n). LeetCode #146.
↓
DOUBLY LLHARD
Q9: Design a browser's back/forward navigation. Which data structure and why?
HINT
You need to go backward and forward. When you navigate to a new URL from middle, forward history is cleared.
ANSWER
Doubly Linked List with a cursor pointer:
• Back = cursor = cursor.prev O(1)
• Forward = cursor = cursor.next O(1)
• New URL = insert after cursor, truncate everything after O(1)

Alternative: Two Stacks — backStack and forwardStack. Both are O(1). DLL is more natural (the history IS a list). Two stacks are simpler to implement.

CIRCULAR LLSENIOR
Q10: Josephus Problem — N people in a circle, every Kth person eliminated. Optimal solution?
HINT
CLL simulation is O(NK). Can you do better with math?
ANSWER
CLL approach: Build circular list. Count K, remove node, repeat. O(NK) time.

Mathematical O(N):
J(1, K) = 0 J(N, K) = (J(N-1, K) + K) % N

After removing the Kth person, the problem shifts to N-1 people with a rotated starting position. The +K accounts for the rotation, %N handles wrap-around.
↓
CIRCULAR LLHARD
Q11: Design a multiplayer game where turns cycle. Players can join/leave mid-game. Singly or Doubly Circular?
HINT
What happens when you need to remove a player and you only have a forward pointer?
ANSWER
Doubly Circular for mid-game join/leave.

Turn cycling: current = current.next → both work fine.

Player leaves: With Singly Circular, finding prev costs O(n). With Doubly Circular: player.prev.next = player.next, player.next.prev = player.prev → O(1).

Player joins after current: 4 pointer rewires → O(1). The prev pointer is the deciding factor.

DESIGNSENIOR
Q12: Design a text editor buffer. Support: insert/delete at cursor, move left/right, undo. Which structure(s)?
HINT
What operation happens most? Typing = insert at cursor. What gives O(1) for that?
ANSWER
Doubly Linked List for character buffer:
• Insert at cursor: O(1) — splice in new node
• Delete at cursor: O(1) — remove node, rewire
• Move cursor L/R: O(1) — prev/next

Array is bad: insert/delete = O(n) shifting.

Undo: Stack of operations. Alternative: Gap Buffer (used by Emacs) — O(1) insert at cursor, O(n) cursor jump. Rope for large files — balanced BST of string chunks.

DESIGNSENIOR
Q13: You need O(1) access by key, O(1) insert at both ends, O(1) delete by reference. Single structure possible?
HINT
No single structure satisfies all three. What combination would?
ANSWER
Doubly Linked List + HashMap.

• HashMap: O(1) key → node lookup
• DLL: O(1) insert at head/tail
• DLL: O(1) delete by node reference

This is the LRU Cache pattern — also Python OrderedDict, LFU Cache, real-time leaderboards, session management with timeout eviction.

ARRAYLISTSENIOR
Q14: Implement remove(value) in O(1) for an unordered ArrayList. How?
HINT
Normal remove is O(n) due to shifting. Can you avoid the shift entirely?
ANSWER
Swap-with-last trick + HashMap:

1. HashMap maps value → index for O(1) lookup
2. To remove value at index i:
  • Swap arr[i] with arr[size-1]
  • Update HashMap for swapped element
  • Decrement size
  • Remove old entry from HashMap

No shifting needed! O(1) for all operations. Trade-off: order is not preserved. Used in game engines for entity removal.

SINGLY LLSENIOR
Q15: Merge two sorted linked lists into one sorted list. Time and space complexity?
HINT
Compare heads of both lists, pick smaller, advance that pointer. Like merge step of merge sort.
ANSWER
dummy = new Node(0), tail = dummy while (l1 != null && l2 != null):   if l1.val <= l2.val:     tail.next = l1; l1 = l1.next   else:     tail.next = l2; l2 = l2.next   tail = tail.next tail.next = l1 ?? l2 return dummy.next

Time: O(m+n), Space: O(1) — reuses existing nodes. The dummy node trick avoids special-casing the first element. LeetCode #21.

LINKED LISTSENIOR
Q16: Merge K sorted linked lists. What's the optimal approach?
HINT
Naive: merge pairs one by one. Better: use a data structure that always gives you the minimum.
ANSWER
Min-Heap approach:
1. Insert head of all K lists into a min-heap
2. Extract min → add to result
3. If extracted node has .next, push it to heap
4. Repeat until heap empty

Time: O(N log K) — N total nodes, each heap op is O(log K). Space: O(K).

Alternative: Divide & Conquer — merge pairs K→K/2→...→1. Same O(N log K). LeetCode #23 (Hard).

DOUBLY LLSENIOR
Q17: Flatten a multilevel doubly linked list where nodes can have child sublists. (LeetCode 430)
HINT
When you encounter a child, splice the child list into the main list. Think recursively or use a stack.
ANSWER
Iterative approach:
curr = head while curr:   if curr.child:     childTail = findTail(curr.child)     childTail.next = curr.next     if curr.next: curr.next.prev = childTail     curr.next = curr.child     curr.child.prev = curr     curr.child = null   curr = curr.next

Splice child list between current and current.next. Each node processed exactly once → O(n) overall.

ARRAYSENIOR
Q18: Implement a circular buffer for a streaming system. 1M events/sec, fixed capacity, overwrite oldest when full.
HINT
Use an array with two pointers (read/write) and modulo arithmetic.
ANSWER
Array-based ring buffer:
capacity = N, arr = new[N] writeIdx = 0, count = 0 write(event):   arr[writeIdx] = event   writeIdx = (writeIdx + 1) % N   count = min(count+1, N) readOldest():   readIdx = (writeIdx - count + N) % N   return arr[readIdx]

All ops O(1). Array is ideal: fixed size, cache-friendly, no allocation overhead. Used in OS kernel logs, network packet buffers, audio processing.

LINKED LISTSENIOR
Q19: Check if a singly linked list is a palindrome in O(n) time and O(1) space.
HINT
Combine three techniques: find middle, reverse second half, compare.
ANSWER
Three steps:
1. Find middle with slow/fast pointers → O(n)
2. Reverse second half in-place → O(n)
3. Compare first half with reversed second half → O(n)
4. (Optional) Reverse second half again to restore

1→2→3→2→1 : mid=3, reverse after → compare 1==1 ✓, 2==2 ✓

O(n) time, O(1) space. LeetCode #234.

SYSTEM DESIGNSENIOR
Q20: In a real system, when would you NEVER use a LinkedList despite its O(1) insert? When is ArrayList always better?
HINT
Think beyond Big-O: hardware, GC, real-world workloads.
ANSWER
LinkedList is almost always worse in practice when:

1. Sequential access patterns — cache misses kill LL performance
2. Small-to-medium sizes (<10K) — ArrayList's shifting is faster due to cache effects
3. GC-heavy environments — each LL node is a separate object → more GC pressure
4. Memory-constrained — 24+ bytes per node vs 4-8 for array

Java's own guidance: ArrayList preferred for most use cases. LinkedList wins ONLY when you have a direct node reference AND frequent removal of that specific node.
